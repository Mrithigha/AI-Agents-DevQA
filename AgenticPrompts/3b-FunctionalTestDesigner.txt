### PHASE 7 — Design functional BDD test cases

## Primary source — the story and Confluence knowledge
Using the ACs from the story AND your full understanding of the application
from Confluence, design scenarios covering:

1. Happy path — primary user journey satisfying each AC
2. Alternative paths — valid variations of the happy path
3. Negative scenarios — invalid inputs, unauthorised access,
   missing data, boundary conditions
4. Edge cases — empty states, session expiry, concurrent actions,
   direct URL navigation without prior state, page refresh behaviour
5. Cross-feature impact — does this story affect any other feature
   described in Confluence? If so write scenarios for that impact too.

## Secondary source — PR diff insights
After completing all story-based scenarios, check the PR diff for:
- Any UI behaviour introduced not covered by the story ACs
- Any error states rendered that should be tested as scenarios

Add these ONLY as additional scenarios tagged: @pr-insight

IMPORTANT: If the PR contradicts the story AC, write scenarios that
validate what the STORY says. Add a comment above the scenario:
  # Discrepancy: PR implements X but story requires Y.
  # Scenario tests story requirement.

## Gherkin rules
- Write from the USER perspective using business language only
- No technical language (no mention of APIs, endpoints, database, DOM)
- Each scenario must be independently executable
- Use Background for pre-conditions shared across all scenarios
- Use Scenario Outline with Examples for data-driven cases
- Every feature file must open with the user story statement:
  As a {role}
  I want to {goal}
  So that {benefit}
- Tags required on every scenario:
  @smoke — critical path scenarios only
  @regression — all scenarios
  @{JIRA-ID} — for traceability
  @pr-insight — for PR-derived scenarios only
